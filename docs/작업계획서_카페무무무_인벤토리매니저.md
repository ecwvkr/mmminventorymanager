## 0. 이 문서의 목적

프롬프트·기능명세서·유저플로우·와이어프레임을 대조해 **실제 개발 착수 전에 확정해야 할 것**(스키마 갭, 범위 결정)을 먼저 정리하고, 그 위에서 화면별·단계별 작업 순서를 정의한다. 애매한 결정을 코드로 밀어붙이지 않고 표면화하는 것이 목적.

## 1. 먼저 확정해야 할 것 (착수 전 의사결정)

프롬프트 스키마와 명세서 요구사항 사이에 **불일치·누락**이 있다. 아래는 코드 짜기 전에 답이 필요한 항목.

### 1-1. 스키마 갭 (프롬프트 `items` 테이블에 필드가 빠져 있음)

| # | 명세서 요구사항 | 프롬프트 스키마 | 조치 |
|---|---|---|---|
| A | 대시보드 "즐겨찾기(pinned)" 리스트 | `items`에 pinned 필드 없음 | `is_pinned boolean default false` 추가 |
| B | 품목별 유통기한 임박 알림일수(7~2일 개별 설정, 명세 4.2.3) | 필드 없음 | `expiration_alert_days int default 7` 추가 |
| C | "발주 완료→입고 확인 시 주문 수량만큼 가산" | 수동 발주의 주문 수량 저장처 없음 (`periodic_order_quantity`는 주기발주용) | `pending_order_quantity numeric` 추가 (발주 시 기록, 입고 시 가산 후 null) |
| D | 거래처 정보 재사용 (명세 3.2.1) | `items`에 `vendor_name/order_url/order_contact` 직접 저장(비정규화) | **MVP는 비정규화 유지 권장.** 거래처 수십 개·정보 반복 편집 니즈 나오면 그때 `vendors` 테이블 분리 |

> **권장:** A·B·C는 명세서 필수 기능이라 반드시 추가. D는 지금 나누지 말 것(YAGNI) — 품목에 문자열로 두고, 재고현황 화면에서 `vendor_name`으로 group by 하면 "거래처별 모아보기"가 성립한다. 정규화는 실제 재사용 고통이 생긴 뒤에.

### 1-2. 범위 결정 — ✅ 확정 (2026-07-08)

| 항목 | 결정 |
|---|---|
| **로그인/인증** | ✅ **공용 태블릿 모드로 시작.** DB에는 계정 스키마(`profiles` 테이블 + Supabase Auth `auth.users`)를 **미리 만들어 두되 로그인 강제는 하지 않음.** 직원 증가 시 로그인 활성화만 하면 되도록 준비. `changed_by`는 당분간 설정의 "현재 작업자" 선택값(localStorage)으로 채움 |
| **구글 드라이브 자동 백업** | ✅ **기능 구현 포함(Phase 1).** OAuth + CSV 생성 + 스케줄 업로드(명세 5.2). Vercel Cron 또는 Supabase Scheduled Function 사용 |
| **거래처 정보** | ✅ **비정규화 유지.** `items`의 `vendor_name/order_url/order_contact` 문자열 그대로, 화면에서 group by |
| **작업자(`changed_by`)** | ✅ 설정에서 "현재 작업자" 선택 → localStorage → 로그 기입. 추후 로그인 도입 시 `profiles.id`로 승격 |
| **나머지 전 항목** | ✅ 계획서 권장안대로 진행 |

## 2. 확정 데이터 스키마 (프롬프트 + 갭 보완)

```sql
-- items
id uuid pk default gen_random_uuid()
name text not null
category text
price numeric default 0
unit text
current_stock numeric default 0
min_required_stock numeric default 0
status text check (status in ('정상','재고 부족','배송중','비활성화')) default '정상'
tags text[] default '{}'
expiration_date date            -- nullable
expiration_alert_days int default 7          -- [갭B]
order_url text                  -- nullable
order_contact text              -- nullable
vendor_name text                -- 비정규화 유지 [갭D]
is_active boolean default true
is_pinned boolean default false              -- [갭A]
image_url text                  -- nullable
memo text                       -- nullable
is_periodic_order boolean default false
periodic_interval_days numeric  -- nullable
periodic_order_quantity numeric -- nullable
pending_order_quantity numeric  -- nullable [갭C] 발주 대기 수량
last_ordered_date timestamptz   -- nullable
updated_at timestamptz default now()

-- inventory_logs
id uuid pk default gen_random_uuid()
item_id uuid references items(id)
action_type text check (action_type in ('입고','출고','실사','상태변경','등록'))
previous_quantity numeric
change_quantity numeric
new_quantity numeric
changed_by text
created_at timestamptz default now()
```

**DB에서 처리(앱코드 대신):**
- `updated_at` 자동 갱신 → `trigger`
- `current_stock <= min_required_stock` 시 `status='재고 부족'` 자동화 → Postgres **trigger** (앱 여러 경로에서 재고가 바뀌므로 트리거 한 곳에 두는 게 정답. 앱코드에 흩뿌리면 누락 발생)
- 일괄 반영·발주완료·입고확인은 **RPC(Postgres 함수)** 로 원자 처리 (재고 변경 + 로그 insert를 한 트랜잭션에)

**계정 스키마 (미사용 준비, §1-2 확정):**
```sql
-- profiles : Supabase Auth(auth.users)와 1:1. 지금은 생성만 하고 로그인 강제 안 함
id uuid pk references auth.users(id)
name text
role text check (role in ('admin','user')) default 'user'
created_at timestamptz default now()
```
- 지금은 로그인 없이 앱 사용(공용 태블릿). RLS는 열어두거나 anon 정책으로 시작.
- 직원 증가 시: Supabase Auth 활성화 + `changed_by`를 문자열 → `profiles.id` 참조로 승격.

**차트 데이터:** 명세 1.4 자산가치 "추이 그래프"는 일별 스냅샷 테이블이 필요하다. MVP는 **현시점 카테고리별 분포 차트**(프롬프트 요구)만 구현하고, 시계열 추이는 Phase 2. (스냅샷 cron 없이는 과거 데이터가 없음 — 없는 데이터로 추이선 못 그림)

## 3. 아키텍처 / 프로젝트 구조

```
app/
  layout.tsx            # 하단 네비 + PWA 메타
  (tabs)/
    page.tsx            # 대시보드
    inventory/page.tsx  # 재고관리(2분할+장바구니)
    register/page.tsx   # 재고등록
    status/page.tsx     # 재고현황+거래처 발주
    settings/page.tsx   # 설정(이력/백업)
  api/ (필요시 route handler; 대부분 Supabase RPC 직접호출)
components/
  BottomNav, ItemCard, StepperModal, CartPanel, ScannerModal, ...
lib/
  supabase.ts          # 클라이언트
  queries.ts           # 조회/RPC 래퍼
public/
  manifest.json, icons/, sw는 next-pwa 자동생성
```

- **상태관리:** 장바구니는 페이지 로컬 상태(useState/useReducer)로 충분. 전역 스토어(zustand 등) 도입 금지 — 장바구니는 한 화면 안에서만 산다.
- **PWA:** `next-pwa` 또는 App Router `manifest.ts` + 직접 sw. 새 의존성 최소화 위해 `manifest.ts`(네이티브) 우선 검토.
- **실시간:** 대시보드 실시간 동기화는 Supabase Realtime 구독. 과하면 화면 진입 시 refetch로 대체 가능(MVP 허용).
- **바코드 스캔:** HTML5 카메라 → 기존 경량 라이브러리(`@zxing/browser`) 사용. 직접 디코더 구현 금지.

## 4. 화면별 구현 계획

프롬프트 구현 가이드 순서(재고관리 → 등록 → 현황/발주 → 대시보드 → 설정)를 따른다. 장바구니 로직이 가장 핵심/위험이라 먼저 친다.

### P2. 재고관리 (최우선)
- 2분할: 좌(카테고리 가로스크롤 바 + 검색 + 스캐너 진입 + 물품카드 그리드) / 우(장바구니 패널)
- 카드 클릭 → **스테퍼 모달**: `입고[+] / 출고[-] / 실사[Set]` 토글 필수 선택 + 수량 → 장바구니 담기
- 장바구니 "적용" → **RPC 일괄 반영** (재고 batch update + logs insert, 원자적). "취소" → 비우기
- 품목 비활성화(`is_active=false`, 로그 `상태변경`)
- 바코드 스캔 실패 시 수동검색 폴백(명세 2.1.4)

### P3. 재고등록
- 폼: 이미지 업로드(Supabase Storage; 없으면 플레이스홀더) · 이름 · 카테고리(동적 생성 허용) · 거래처명 · order_url/contact · 초기재고 · 단위 · 가격 · 최소보유수량 · 다중태그 입력칩 · 유통기한(`<input type=date>`, 라이브러리 금지) · 유통기한 알림일수 · 주기발주 토글+주기일수+수량
- 저장 시 `등록` 로그 생성, 재고부족 판정은 DB 트리거가 처리
- 카테고리 동적 생성: 기존 카테고리 distinct 조회 + 자유 입력

### P4. 재고현황 + 거래처별 일괄 발주
- 전체 품목 테이블: 상태 필터('재고 부족/배송중/비활성화') + 정렬
- **거래처 그룹 탭:** `status='재고 부족'` 품목을 `vendor_name` group by → 아코디언
  - 거래처별 "웹샵 이동"(order_url) / "전화"(tel: order_contact)
  - 품목별 **"발주 완료"**: RPC로 `status='배송중'`, `last_ordered_date=now()`, `pending_order_quantity` 기록
  - **"입고 확인"**: RPC로 `current_stock += pending_order_quantity`, `status='정상'`, `pending_order_quantity=null`, `입고` 로그

### P1. 대시보드
- 요약 위젯: 활성 품목수 · 재고부족 수 · 총자산가치(Σ current_stock×price)
- 차트: 카테고리별 분포(**CSS 바 우선**, Recharts는 정말 필요할 때만 — 새 의존성)
- **주기발주 알림:** `now - last_ordered_date >= periodic_interval_days` → 품목명 + periodic_order_quantity 강조 표시
- 바로가기: 즐겨찾기(`is_pinned`) 리스트 + 발주필요 리스트, 클릭 시 상세

### P5. 설정
- 이력조회: `inventory_logs` 타임라인 (누가/언제/무엇/어떻게), 필터·페이지네이션
- 백업: 수동 CSV 다운로드(재고현황 + 로그) **+ 구글 드라이브 자동 백업**(§1-2 확정)
  - 구글 OAuth 연결 → 대상 폴더 지정 → 주기(일/주/월)·시간 설정 → CSV 자동 업로드 → 백업 로그/상태 표시 (명세 5.2~5.3)
  - 스케줄러: **Vercel Cron**(`vercel.json`) → route handler에서 CSV 생성 후 Drive API 업로드. 리프레시 토큰은 Supabase에 저장
- 작업자 선택(공용 태블릿 모드): 설정에서 현재 작업자 지정 → localStorage → 로그 `changed_by`에 기입

### 공통
- 하단 네비 5탭 · PWA manifest/아이콘/설치 · 반응형(모바일↔태블릿) · Lucide 아이콘

## 5. 단계별 마일스톤 (WBS)

| 단계 | 내용 | 산출물/DoD |
|---|---|---|
| **M0 셋업** | Next.js+Tailwind+PWA 골격, Supabase 연결, 하단네비, manifest | 홈화면 추가·전체화면 실행 확인 |
| **M1 DB** | 스키마 마이그레이션 + 트리거(재고부족 자동/ updated_at) + RPC(일괄반영·발주·입고) | Supabase에서 RPC 단위 검증 |
| **M2 재고관리** | 2분할·스테퍼·장바구니·일괄반영·스캔 | 담기→적용 시 재고+로그 정확히 반영 |
| **M3 등록** | 폼·이미지·동적카테고리·주기발주 | 등록 즉시 목록·상태 반영 |
| **M4 현황/발주** | 필터테이블·거래처그룹·발주완료·입고확인 | 발주→배송중→입고→재고가산 왕복 검증 |
| **M5 대시보드** | 위젯·분포차트·주기알림·바로가기 | 수치 실시간/재진입 갱신 |
| **M6 설정** | 이력 타임라인·CSV 다운로드·계정 스키마(미사용)·작업자 선택 | 로그 필터·CSV 정상 |
| **M7 백업** | 구글 OAuth·Drive 업로드·Vercel Cron 스케줄·백업 로그 | 지정 폴더에 CSV 자동 업로드 확인 |
| **M8 마감** | 반응형·PWA·Vercel 배포·QA | 태블릿/모바일 실기기 확인 |

각 단계 **DoD: 모든 재고 변경이 `inventory_logs`에 정확한 이전/변동/이후 수량으로 남는지** 확인(프롬프트 4단계 요구).

## 6. 검증 포인트 (놓치면 데이터 오염되는 곳)

- 일괄 반영·발주완료·입고확인은 **반드시 트랜잭션(RPC)** — 재고만 바뀌고 로그 누락 / 로그만 남고 재고 안 바뀜 방지
- 재고부족 자동판정은 **트리거 한 곳** — 등록/입고/출고/실사/입고확인 모든 경로가 트리거를 거치게
- `실사(Set)`: change_quantity = new - previous 계산 정확성
- 입고확인 시 `pending_order_quantity` null 방어(중복 클릭 시 이중 가산 금지)
